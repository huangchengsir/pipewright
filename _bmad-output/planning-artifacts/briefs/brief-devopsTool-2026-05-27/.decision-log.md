# Decision Log — brief-devopsTool-2026-05-27

规范化记忆与审计轨迹。每个决定、变更、推翻都记录在此。

## 2026-05-27

- **意图确认**:Create(全新项目,无现成材料)。
- **初始 brain dump(用户)**:
  1. 问题:个人开发者部署多个服务,部署复杂。
  2. 形态:类似 Jenkins 的平台。
  3. 现有材料:无。
  4. 目标:做成开源工具。
- **研究子智能体结论(landscape)**:赛道极度拥挤。Coolify(56K★)几乎正面覆盖"个人开发者轻松部署多服务"。其它:Portainer 37K、Dokploy 34K、Dokku 31K、Jenkins、CapRover 15K、Kamal 14K。
  - 已解决:push-to-deploy。
  - 仍未解决的痛点 / 潜在缺口:单机即单点故障;隐形运维债(备份/密钥轮换/零停机);Docker 磁盘膨胀;团队协作/RBAC薄弱;监控告警需另装;Coolify 功能多但更新后 UI 不稳。
- **重大澄清(用户,第2轮)**:产品定位**不是 Coolify 类 PaaS**,而是 **Jenkins 类 CI/CD + 部署编排工具**。
  - **动机 = 方向2(被现有工具坑过)**。Jenkins 三宗罪:① 内存占用太大;② UX 不友好、不够直观简洁(插件虽丰富但不简洁);③ 没有 AI 能力。
  - **核心需求/差异化候选**:体积小、内存占用小;一个平台可打包多套不同开发环境的镜像;远程部署到不同服务器;多服务器管理;内置 AI。
  - 用户称在 GitHub 上没找到符合需求的工具(待研究子智能体核实轻量级 CI 赛道:Woodpecker/Drone/Concourse 等是否已覆盖)。
- **待拷问**:① AI 到底做什么具体的活(防止 buzzword);② 范围会不会过大(构建+部署+多机管理+AI 对一个轻量工具是不是太多)。

- **研究子智能体结论(轻量 CI 赛道,第2轮)** —— 强力验证了用户直觉:
  - 真·轻量仅 Woodpecker(50–100MB)、Drone(~50MB)、Laminar(<20MB);Jenkins 500MB+。**硬约束:用 Go,不用 Java。**
  - **没有任何工具同时具备**:小体积 + 干净 UI + 多环境镜像构建 + 原生多服务器部署 + 多服务器管理。现实是 3 工具拼装(Woodpecker/Drone + Ansible/Kamal + Portainer)。→ **差异化 = 一个工具替掉三个 + 轻量 + AI。**
  - **AI 两个阵营**:脚手架/生成配置(红海:ciraft、Pipe Pilot 等)vs 运行时智能(空地:仅 ForgeAI 这个 Jenkins 插件 ~100 用户)。
- **用户回答(第3轮)**:
  1. AI = 智能配置 + 运行时智能运维监控。【教练提示:智能配置偏红海,运行时监控是空地 —— 待澄清"智能配置"具体指什么】
  2. **一个人做,MVP 起步,慢慢迭代开源,先把特点打出去**。
  3. 工作模式 = **教练道**。
- **张力(需在 Scope 章解决)**:单人 + MVP vs "构建+多机部署+多机管理+2类AI" = 4个产品的工作量。必须切出一个最小楔子。

- **重大需求澄清(第5轮)**:「多套环境」≠ dev/staging/prod,而是**多版本工具链隔离构建**(前端 Node 14 vs 22;Java JDK 8/11/17/23),且**绝不污染宿主机**。→ 架构定为**容器内隔离构建**(每项目版本钉死的容器,build 完即焚)。
- **需求(第6轮)**:平台本体须**同时支持 Docker 运行 + 本地编译原生运行**。
- **架构决定**:MVP 部署 = **无 agent / 纯 SSH**(目标机仅需 SSH+Docker)。agent(内网穿透+实时监控)推迟到 **v2**,配合 ⑥;形态为独立进程经网络协议通信(非 in-process 插件)。
- **语言选型 = Go(已锁定,第7轮)**。触发点:作者声明「AI 写代码、自己不深读代码、不以个人 Java 背景为约束」→ Node 的唯一优势作废;而轻量/不污染/本地编译/双运行模式条条对齐 Go 单静态二进制。备选 Node/TS(fallback)、Java(否决:JVM 重)、Rust(否决:单人慢)。详见 addendum。
- **风险登记**:作者「不深读代码」对开源项目的维护/安全/社区 PR 审阅是长期风险 → 记入 addendum,留待 PRD/架构。
- **Scope(第一版)**:v1 = ①②③④⑤ + 双运行模式;v2 = ⑥ + 可选 Go agent;明确不做 = HA/集群、RBAC、服务市场。

- **新增需求(第8轮)**:⑦ Webhook 触发 + 分支感知(多代码平台);⑧ 平台内部代码管理 + diff;⑨ 构建通知多平台×多方式。
- **作者决定(第9轮):推翻 MVP 极简策略 —— 不再分 v1/v2 最小切片,直接做完整产品。** 理由:可"多开几个 AI 开发 agent 并行",单人产能不再是约束。
  - **教练异议(已记录,作者保留决定权)**:AI 加速的是写代码,非「20 个生产服务稳定运营」(真实日历时间,不可并行压缩);范围越宽缝合面越多,叠加作者「不深读代码」放大集成/连贯性风险;依赖顺序绕不过。
  - **折中(作者方向下保留的唯一纪律)**:**dogfooding 优先的落地排序** —— 先打通端到端竖切(1平台webhook→隔离构建→部署1台→通知1渠道),再横向铺开。写入 Scope。
- **确认(第10轮)**:⑥ 运行时AI + 可选 Go agent **并入本期目标范围**。端到端竖切首个目标:**代码平台 = Gitee 码云;通知渠道 = 企业微信/钉钉(首)、飞书、邮件**。Scope 已据此定稿。

- **作者澄清审核策略(第13轮)**:非不审 —— v1/v2 快速上线后按 PR + 全量安全评审,仅起步不关注。→ 风险由致命降为可控。教练 caveat:审核闸口绑"外部用户接入"时点而非版本号;SSH/凭据敏感面建议 v1 即审。已记入 addendum 风险章节。
