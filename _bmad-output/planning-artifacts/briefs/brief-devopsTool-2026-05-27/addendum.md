# Addendum — Pipewright 简报附录

保留属于下游文档(PRD / 架构 / 方案设计)的深度内容与技术约束,不挤占简报正文。

## 技术选型决策:Go(已锁定)

**决定**:平台主体使用 **Go**;未来 v2 的可选 agent 同样用 Go。

**理由(需求逐条对齐)**:
- 头号卖点「轻量」:Go 常驻内存 ~10–50MB,对标 Woodpecker;Jenkins 500MB+ 出局。
- 「不污染宿主机」+「本地编译原生运行」+「Docker 双运行模式」:Go 单静态二进制是教科书级契合 —— 同一份产物,本地拷贝即跑(零依赖、零污染),或塞进 alpine/scratch 打成几 MB 镜像。
- 调度 Docker 多版本隔离构建:官方 Docker SDK 即 Go。
- 同赛道正统:Woodpecker、Drone、Docker、K8s、Portainer 皆 Go。
- 作者决定「由 AI 编写代码、自己不深读代码、不以个人语言背景为约束」→ Node 的唯一优势(作者 Java 背景上手快)被作废,Go 成为更对齐的选择。

**备选与否决理由**:
- **Java**:JVM 重量直接违背核心承诺;GraalVM Native 虽可压到几十 MB,但反射配置/构建复杂度对单人项目摩擦过大。否决。
- **Node/TypeScript**:足够轻于 Jenkins(~80–150MB,Dokploy 34K★ 为证),AI/LLM SDK 生态最成熟;但本地编译原生运行需借 Bun/pkg 打包(工具链较糙、二进制较大),且裸跑需装 Node(污染宿主机)。作为 fallback 保留,非首选。
- **Rust**:更轻更省,但单人上手与迭代速度较慢。否决。

## 部署架构

- **多版本隔离构建**:每个项目在版本钉死的容器内构建(`node:14`/`node:22`、`eclipse-temurin:8`/`:11`/`:17`/`:23` 等),构建结束后销毁容器。宿主机仅需 Docker,工具链不落地、不冲突、不污染。
- **MVP 部署 = 无 agent**:中控机经 SSH 推送到目标机;目标机仅需 SSH + Docker。优点:零安装、加减机器轻松。代价:够不到无法 SSH 入站的内网机器;实时监控靠轮询。
- **v2 可选 agent**:配合 ⑥ 运行时监控引入。形态 = **独立进程 / 独立二进制,经 gRPC / HTTP / WebSocket 与主平台通信**(参考 Drone server+agent),**不是 in-process 插件**。买点:内网穿透(agent 主动拨出)、实时指标/日志流、本地自治、缩小中控信任面。

## 可行性判断与风险(2026-05-27 教练评估)

**按作者自定义成功标准(自用 20 个服务 + 100 star):大概率可达。** 这是执行/坚持问题,非可行性问题;技术路线(Go + 容器隔离构建 + SSH 部署)成熟,无科研难题。
**作为爆款开源(数千用户、在 Coolify/Woodpecker 夺食):逆风明显。** 障碍:① 赛道挤(Coolify 56K★、Dokploy 34K★……获客难);② 单人维护 + 代码审核滞后带来的信任/安全顾虑(工具持有目标机 SSH 权限,攻击面大);③ 范围大、单人易停在半成品;④ 差异化须真正落在「运行时 AI 运维」这块空地,否则泯然于众多轻量 CI。
**利好(真实牌面):** 真实痛点→真实产品;缺口经两轮研究验证(整合 + 运行时 AI 确无人做);成功标准朴素可达;AI 开发产能为真实力量倍增器;"作者用它跑 20 个生产服务"是天然开源叙事。

### 代码审核策略(作者澄清)
并非不审。计划 = **v1/v2 快速上线 → 之后按 PR 审核 + 全量安全评审**;仅最初阶段不关注。该策略将"AI 写码、起步不审"的风险从致命降为可控。

**教练补充 caveat(留待 PRD/架构):**
- 审核闸口应绑定 **「外部用户开始拿它连自己的服务器」** 这一时点,而非单纯"v1/v2 做完"的日历点 —— 陌生人一旦托付凭据,未审代码即成其风险。
- **SSH / 密钥 / 凭据处理**这一安全敏感面,建议即便 v1 也纳入审查(事后全量补审比增量审更难)。
- 作者宜保持"能读懂、能判断"的代码水平,实现细节可交 AI。
