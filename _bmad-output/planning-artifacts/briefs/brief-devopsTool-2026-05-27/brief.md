# Product Brief: Pipewright

> 轻量、AI 原生、一体化的自托管 CI/CD 与部署编排平台。技术深挖、选型理由与风险见同目录 `addendum.md`。

## Executive Summary

Pipewright 是一个**轻量级、自托管、开源**的 CI/CD 与部署编排平台,用 **Go** 实现。它把今天需要拼三个工具(CI 工具 + Ansible/Kamal + Portainer)才能完成的事 —— **多版本隔离构建、多服务器远程部署、多服务器管理** —— 收进一个工具,并把 AI 做进配置与运维;平台自身既可 Docker 运行,也可本地编译为原生程序运行。

它源自一个真实痛点:作者用 Jenkins 管理自己约 20 个服务与网站时,苦于它的臃肿(JVM 500MB+)、繁琐的 UX 与缺失的 AI,而现成工具中没有一个**同时**满足「轻量 + 多版本隔离构建 + 不污染宿主机 + 多机部署与管理」。

**为什么是现在**:轻量 CI 已被 Go 系工具验证可行(Woodpecker、Drone),但没有一个把构建、部署、管理整合到一起,更没有一个把 AI 做进运行时。Pipewright 的成功标准朴素而硬核 —— 作者能用它稳定承载自己全部 20 个生产服务,并以此作为开源叙事的起点。

## The Problem

源自作者使用 Jenkins 的真实痛点:
- **太重**:基于 JVM,内存 500MB+ 起步。
- **难用**:UX 不直观、不简洁;插件虽多但不成体系。
- **没有 AI**:配置全靠人工,失败排查全靠翻日志。
- **能力割裂**:「多版本隔离构建 + 多服务器远程部署 + 多服务器管理」现实中得拼三个工具(CI 工具 + Ansible/Kamal + Portainer)。
- **多版本工具链冲突 + 宿主机污染**:同一平台需支持多个版本的工具链(前端 Node 14/22,Java JDK 8/11/17/23 等),而在宿主机直接安装会造成环境污染与版本冲突。

## The Solution

一个轻量级、自托管、开源的 CI/CD + 部署编排平台,一个工具完成构建、部署与管理,并内置 AI:
- **多版本隔离构建**:每个项目在版本钉死的容器内构建(`node:14`/`node:22`、`eclipse-temurin:8`/`:23` 等),构建结束后销毁容器,**宿主机零污染**。
- **平台双运行模式**:既可作为 Docker 容器运行,也可本地编译为原生程序直接运行(不依赖 Docker)。
- **远程部署**:首阶段经 SSH 推送到目标机,目标机只需 SSH + Docker;后续可选轻量 agent 用于内网穿透与实时监控。
- **技术栈**:Go(平台主体与未来 agent 均采用 Go)。选型理由与备选见 `addendum.md`。

## What Makes This Different

核心差异化 = **一个工具替掉三个 + 轻量 + AI**。经竞品研究验证:没有任何现有工具**同时**具备「轻量 + 多版本隔离构建 + 多机部署 + 多机管理 + AI」。
- **轻量**:目标常驻内存对标 Woodpecker(约 50–100MB),远低于 Jenkins(500MB+);Go 单静态二进制。
- **整合**:省掉「CI 工具 + Ansible/Kamal + Portainer」的三件套拼装。
- **AI(两条腿)**:① 生成贴合本平台格式的流水线配置,降低迁移门槛;② 运行时智能运维(读日志定位失败根因、异常检测告警)。后者目前是市场空地,也是最锋利的差异点。

> 竞品坐标:Coolify 56K★(PaaS,非同类)、Woodpecker ~6.5K★(轻量 CI,缺多机部署 UX)、Drone ~31K★(被 Harness 收购前景不明)、Kamal/Ansible(只管部署)、ForgeAI(仅 Jenkins 的运行时 AI 插件)。

## Who This Serves

- **主要**:像作者一样的个人开发者 / 独立开发者 —— 自管数台服务器、运行多个服务,嫌 Jenkins 过重、体验繁琐,又不愿拼装多个工具。
- (待验证是否延伸到 5–50 人小团队 / homelab 用户)

## Success Criteria

**北极星(主信号):作者自用且稳定运营。** 作者把自己约 **20 个服务与网站**全部迁移到 Pipewright 上稳定运营,使其成为日常部署的唯一工具、彻底替代 Jenkins。「连作者都把 20 个生产服务托付给它」既是成功判定,也是最硬的开源叙事。

里程碑:
- **M1**:自己 1–2 个服务跑通,日常在用。
- **M2**:20 个服务全部迁移,稳定运营,不再频繁救火。
- **M3(被看见)**:GitHub **100 star**(作者认为达到即"不错")。
- **M4(被陌生人用)**:出现非作者的真实用户 / issue。

## Scope

### 目标范围
作者决定借助多个 AI 开发 agent 并行,不以单人产能为约束,直接做完整产品。全部纳入:
- ① 轻量内核 + 干净 Web UI
- ② **Webhook 触发部署 + 分支感知**(push 对应分支触发对应环境;多代码平台)
- ③ **多版本隔离构建**(容器内,Node 14/22、JDK 8/11/17/23 等任意版本并存,宿主机零污染)
- ④ **多服务器远程部署**(无 agent,中控机经 SSH 推送)
- ⑤ **多服务器管理**(经 SSH 查看日志 / 重启 / 状态)
- ⑥ **平台内部代码管理 + diff**(clone / 管理源码、查看变更)
- ⑦ **构建通知**(多平台 × 多方式)
- ⑧ **AI · 生成贴合本平台格式的流水线配置**
- ⑨ **AI · 运行时智能运维**(读失败日志定位根因、异常检测告警)
- ⑩ **可选轻量 Go agent**(配合 ⑨:内网穿透、实时监控)
- ⑪ **平台双运行模式**(Docker / 本地编译原生)

### 落地排序(dogfooding 优先)
不是因产能受限,而是为尽快得到"能真跑"的东西。先打通一条**端到端竖切**:**Gitee** push webhook(分支感知)→ 隔离构建 → 部署到 1 台服务器 → **企业微信/钉钉**通知,让作者第 1 个服务真正跑起来;再横向铺开其它代码平台(GitHub/GitLab/Gitea)、多机、diff、其它通知渠道(飞书、邮件)、运行时 AI 与 agent。

### 明确不做
- HA / 集群、企业级 RBAC、一键服务市场 —— 守住"小而锋利",不长成又一个 Coolify。

## Vision

**远期(2–3 年)**:成为独立开发者与小团队在「嫌 Jenkins 过重、又不想拼装 CI 工具 + Ansible + Portainer 三件套」时,默认会拿起的那个**轻量、AI 原生、一体化**的自托管部署平台。其中,AI 从「帮你写配置」走向「帮你运维」是这条路上的关键一跃。坚持边界:不做 HA/集群、企业 RBAC、服务市场,守住"小而锋利"。

(近期与中期目标分别见 Success Criteria 与 Scope。)
