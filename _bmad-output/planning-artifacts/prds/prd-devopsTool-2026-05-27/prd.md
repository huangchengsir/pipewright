# PRD: Pipewright

## 0. 文档目的

本 PRD 面向作者本人(单人开发,借多 AI 开发 agent 并行实现)及未来的开源贡献者。它基于产品简报 `../briefs/brief-Pipewright-2026-05-27/`(brief.md + addendum.md)展开,不重复其中已有内容;**竞品定位与差异化论据、技术选型、战略风险姿态见简报正文与 addendum**,架构细节见后续架构文档。结构:术语表锚定词汇,能力按 Feature 分组、FR 全局编号嵌套,假设以 `[ASSUMPTION]` 内联标注并在文末索引。

## 1. Vision

Pipewright 是一个轻量、自托管、开源的 CI/CD 与部署编排平台,用 Go 实现。它把开发者今天必须拼三个工具(CI 工具 + Ansible/Kamal + Portainer)才能完成的事 —— 从代码触发,到多版本隔离构建,到多服务器部署与管理 —— 收进一个工具,并把 AI 做进配置与运维。

它服务于像作者这样自管数台服务器、运行多个服务的个人开发者:嫌 Jenkins 过重(JVM 500MB+)、体验繁琐、且毫无 AI,又不愿为凑齐能力去拼装一堆工具。Pipewright 的目标是成为他们日常部署的唯一工具。

衡量它成败的标准朴素而硬核:作者能用它稳定承载自己全部约 20 个生产服务。这既是产品的验收线,也是它面向开源世界最有说服力的叙事。

## 2. Target User

### 2.1 Jobs To Be Done
- 改完代码 push 上去后,**无需手动操心**,服务自动在对的服务器上更新。
- 跑不同版本工具链的项目(Node 14 与 22、JDK 8 与 23),**互不冲突,也不污染宿主机**。
- 在**一个地方**查看所管多台服务器的状态/日志、重启服务,而不必逐台 SSH。
- 构建/部署成功或失败时,**第一时间在企业微信/钉钉等渠道收到通知**。
- 构建失败时,**AI 直接指出问题所在**,而非自己翻一堆日志。

### 2.2 Non-Users (v1)
- 需要高可用/集群、企业级 RBAC/多租户的团队。
- 想要"一键服务市场"式 PaaS(Coolify 那类)的用户。
- 非技术用户(本工具假定用户懂 Docker、SSH、基本运维概念)。

*(5–50 人小团队 / homelab 用户是否纳入,待 v1 自用验证后再议。)*

## 3. Glossary

*下游工作流与读者须严格使用以下术语;PRD 中不得引入同义词。*

- **项目(Project)** — 一个被纳管的代码仓库及其部署目标的集合。每个项目拥有一份流水线配置。
- **流水线配置(Pipeline Config)** — 项目的声明式配置,定义触发事件、分支→环境/服务器映射、构建步骤与部署步骤。可由 AI 生成(见 §4.2)。
- **流水线运行(Pipeline Run)** — 由一次触发产生的「构建 + 部署」执行实例。
- **环境(Environment)** — 一组部署目标的逻辑分组(如 prod / test),由分支映射决定。
- **目标服务器(Target Server)** — 部署目标主机;平台经 SSH 连接,主机需具备 SSH + Docker。
- **凭据(Credential)** — 安全存储的密钥/令牌(仓库访问、服务器 SSH 等),由配置按引用使用,不以明文出现在日志或界面。
- **构建容器(Build Container)** — 为单次构建启动的、版本钉死的隔离容器;构建结束后销毁。
- **构建产物(Build Artifact)** — 一次构建的产出,类型可配置(Docker 镜像 / JAR / dist 静态目录等)。
- **镜像仓库(Registry)** — 外部镜像存储(阿里云 ACR / Docker Hub / Harbor 等);平台不自带,按配置推送。
- **服务(Service)** — 部署在目标服务器上、由平台纳管的运行实例(容器或进程)。
- **容器(Container)** — 服务的一种运行形态(Docker 容器);进容器执行命令(FR-18)等操作仅适用于容器形态的服务。
- **服务器分组(Server Group)** — 目标服务器的逻辑分组(按环境/项目),用于筛选与组织。
- **通知渠道(Notification Channel)** — 接收平台通知的目标:企业微信 / 钉钉 / 飞书 / 邮件 / 自定义出站 webhook。

## 4. Features

### 4.1 代码接入与触发
**描述**:平台为每个**项目**接收代码平台(首发 Gitee)的 webhook;由**流水线配置**声明哪些事件、哪些分支触发何种动作,并把分支映射到**环境 + 目标服务器**。平台以配置的**凭据**拉取(可含私有)仓库,在触发的 commit 上构建,并提供只读代码浏览与部署间 diff。实现 JTBD「push 后自动更新」。

**Functional Requirements:**

#### FR-1: Webhook 接入与事件触发
每个项目拥有独立 webhook 接收端点;触发哪些事件由流水线配置声明,v1 支持 `push`、`tag`、`PR/MR`。亦支持在 Web UI 手动触发一次运行(指定分支/commit)。

**Consequences (testable):**
- 配置 push 触发后,投递一个匹配分支的 push 事件 → 在数秒内创建一次流水线运行。
- 投递未配置/不匹配的事件 → 不创建运行,并记录被忽略的原因。
- Web UI 手动触发指定分支/commit → 创建一次等效的流水线运行。

#### FR-2: 分支 → 环境/服务器 映射
流水线配置维护「分支(支持通配)→ 环境 + 目标服务器集合」的映射。一个分支可映射到多台目标服务器;不同分支可映射到不同环境。

**Consequences (testable):**
- 配置 `main`→prod(机A/机B)、`dev`→test(机C);push `main` → 仅对 A、B 执行 prod 流程。
- push 未映射分支 → 无部署动作(可选择记录或忽略)。

#### FR-3: 仓库凭据配置
用户可为项目配置拉取仓库所需凭据(deploy key 或访问令牌);平台安全存储,拉取时按引用使用。

**Consequences (testable):**
- 配置有效令牌 → 私有仓库克隆成功。
- 凭据无效/缺失 → 运行以明确的"凭据错误"失败,且日志/界面不出现凭据明文。

#### FR-4: 代码管理与 diff
平台在触发 commit 上克隆/更新仓库用于构建;提供**只读**代码浏览,以及 **diff 视图**(本次部署相对上次成功部署):同时呈现 **commit 区间**与**文件级逐行**变更。

**Consequences (testable):**
- 连续两次部署 → diff 视图列出两次间的变更 commit,并可下钻到文件级逐行差异。
- 首次部署 → diff 视图提示"无基线"。
- 代码浏览为只读;界面不提供编辑或提交入口。

**Notes:** 内嵌代码浏览/ diff 的具体实现(如 code-server / Monaco)见 addendum;PRD 仅约束"只读浏览 + 双粒度 diff"能力。

### 4.2 流水线与隔离构建
**描述**:项目的构建由流水线配置驱动,在版本钉死的隔离容器内执行,宿主机零污染;支持自带 Dockerfile 与平台按选定工具链构建两种模型,产出多种类型产物,镜像推送到用户配置的外部仓库。AI 可基于代码仓库生成整份流水线配置。实现 JTBD「多版本工具链互不冲突、不污染宿主机」。

**Functional Requirements:**

#### FR-5: 多版本隔离构建
每次构建在**构建容器**中执行,宿主机零污染,结束即销毁。支持两种输入模型:**(A)** 用户自带 Dockerfile,平台执行 `docker build`;**(B)** 用户在流水线配置选定工具链版本(如 `node:14`、`jdk:23`),平台在对应构建容器内执行构建命令,无需 Dockerfile。

**Consequences (testable):**
- 项目X配 `node:14`、项目Y配 `node:22` → 分别构建成功且互不影响;宿主机未安装任何 node。
- 构建结束后构建容器被销毁;不在宿主机留存工具链。
- 模型 A 与模型 B 均可产出可部署的构建产物。

#### FR-6: 多类型构建产物
构建产物类型可配置:**Docker 镜像 / JAR / dist(静态目录)等**。每种产物可被 §4.3 部署阶段消费。

**Consequences (testable):**
- 配置产出 Docker 镜像 → 产出镜像;配置 JAR → 产出 jar 文件;配置 dist → 产出静态目录。
- 产物类型与后续部署方式对应(镜像→run 容器;JAR→运行;dist→静态服务,见 §4.3)。

#### FR-7: 外部镜像仓库集成
**平台不自带 registry**;镜像产物推送到用户配置的外部**镜像仓库**(阿里云 ACR / Docker Hub / Harbor 等),凭据按 FR-3 管理。非镜像产物(JAR/dist)不进 registry,其传输在 §4.3 处理。

**Consequences (testable):**
- 配置 Harbor + 有效凭据 → 构建镜像成功推送到该 Harbor。
- 凭据错误/仓库不可达 → 推送以明确错误失败,日志不泄露凭据明文。

#### FR-8: AI 生成流水线配置
初始化项目时平台拉取代码;AI **分析仓库**(技术栈/版本/结构)推荐一份流水线配置,用户可用自然语言补充。生成结果以**预览**呈现,支持**一键全部接受**或**逐项部分接受**。

**Consequences (testable):**
- 对 Node 22 仓库初始化 → AI 推荐含 `node:22` 构建的配置。
- 用户补充"push main 部署到生产" → 预览反映该分支映射。
- 用户只勾选部分项接受 → 仅被勾选项写入流水线配置。

#### FR-9: 流水线配置合法性校验
保存流水线配置(无论手写还是 AI 生成)时,平台执行基本合法性校验:schema 正确、必填项齐全、引用的凭据与目标服务器存在。

**Consequences (testable):**
- 配置缺少必填项或引用不存在的凭据/服务器 → 阻止保存并定位到具体错误项。
- 校验通过 → 配置可保存并用于触发流水线运行。

### 4.3 多服务器部署
**描述**:平台经 SSH 把构建产物送达**目标服务器**并启动,按产物类型采用对应运行方式;部署策略针对普通 Docker/SSH 目标做备份/回滚 + 健康检查门控,平台不重复造复杂编排(**k8s 委托延后出 v1**,见 §5/§6)。实现 JTBD「push 后自动在对的服务器上更新」。

**Functional Requirements:**

#### FR-10: 多产物部署执行(经 SSH)
平台经 SSH 将构建产物部署到目标服务器并启动,按产物类型:
- **Docker 镜像**:目标机从镜像仓库拉取 + `docker run`。
- **JAR**:传输到目标机后运行,支持 `java -jar` 直接运行或注册为 systemd 服务(可配)。
- **dist 静态**:传输到目标目录,由 nginx 托管;平台可选地管理 nginx 站点配置(可开可关)。

**Consequences (testable):**
- 镜像产物 → 目标机拉取并 `docker run`,服务可访问。
- JAR 产物 → 按所选方式(直接 / systemd)在目标机运行。
- dist 产物 → 落到配置目录;启用 nginx 管理时,平台写入/更新对应站点配置。

#### FR-11: 部署策略(零停机 + 回滚)
v1 仅针对普通 **Docker/SSH** 目标:保留上一版本(镜像 tag / JAR 备份等);采用"先起新版 → 健康检查通过 → 再切换(端口约定)→ 确认后才停旧版"的方式实现零停机切换,失败时回滚到上一版本。**k8s 委托延后出 v1**(见 §5/§6)。

**Consequences (testable):**
- 普通目标:新版本通过健康检查后切换;失败则保留/恢复旧版本对外可用。
- `[ASSUMPTION]` 普通目标默认保留上一版本 1 份用于回滚。

#### FR-12: 部署健康检查
部署后按配置执行健康检查(HTTP 探针 / 端口存活);检查结果决定切换或回滚。健康检查可选、可配置。

**Consequences (testable):**
- 配置了健康检查 → 新版本不健康则不切换并触发回滚。
- 未配置健康检查 → 容器/进程启动成功即视为部署成功。

#### FR-13: 并行部署与独立结果(非事务性)
多个流水线/任务并行执行(各流水线对应不同服务,**不做跨流水线事务**)。单个流水线向多台目标服务器部署时各机并行、结果独立:一台失败不阻断其余,且失败机按 FR-11 自行回滚,不连累成功机。界面汇总每台状态。

**Consequences (testable):**
- 同时触发多个流水线 → 并行执行,互不阻塞。
- 1 个流水线推 3 台、1 台失败 → 其余 2 台照常完成;失败机回滚;界面显示 3 台各自状态。
- 部分目标失败时,流水线运行整体状态标记为「部分失败」,并触发 FR-20 的部署失败事件(携带失败目标清单)。

### 4.4 服务器与服务管理
**描述**:在统一界面纳管多台目标服务器,按需查看状态与日志、对服务执行操作,免去逐台 SSH。本节为**按需查看 + 手动操作**(经 SSH 拉取,非常驻);**持续实时监控与异常告警见 §4.6**。实现 JTBD「在一个地方看多机状态/日志、重启服务」。

**Functional Requirements:**

#### FR-14: 目标服务器登记与分组
用户登记目标服务器(host/port + 选择 SSH 凭据),平台校验连通性(SSH + Docker 可用);服务器可按**服务器分组**(环境/项目)组织。

**Consequences (testable):**
- 添加服务器并填有效 SSH 凭据 → 校验通过并标记可用;凭据/网络错误 → 标记不可用并提示原因。
- 服务器可归入分组,并按组筛选查看。

#### FR-15: 多机状态总览
统一面板查看所管服务器的**服务器层**(CPU / 内存 / 磁盘)与**服务/容器层**(运行中的服务、状态、端口)。

**Consequences (testable):**
- 面板列出每台服务器的 CPU / 内存 / 磁盘使用。
- 展开某服务器 → 列出其上运行的服务/容器、状态与端口。

#### FR-16: 日志查看(实时 + 历史)
查看服务的日志,支持实时 tail 与历史回看。

**Consequences (testable):**
- 选定一个服务 → 实时 tail 其日志输出。
- 可按时间范围回看历史日志。

#### FR-17: 服务操作(重启 / 停止 / 启动 / 重新部署)
经界面对目标机上的服务执行重启、停止、启动、重新部署。

**Consequences (testable):**
- 重启某服务 → 该服务在目标机重启并反馈结果。
- 触发重新部署 → 等效于重跑该服务的部署阶段(§4.3)。

#### FR-18: 容器内命令执行(交互终端)
提供进入目标机容器执行命令的交互式终端(经 SSH → `docker exec`)。

**Consequences (testable):**
- 对运行中容器打开终端 → 可执行命令并看到输出。
- 该能力为安全敏感操作(生产机远程命令执行),须受凭据/权限约束并可审计 —— 见 §安全 NFR。

### 4.5 构建与部署通知
**描述**:在构建/部署生命周期的关键事件上,经多种**通知渠道**发出通知;事件→渠道映射按项目细粒度配置,通知内容可自定义。实现 JTBD「成功/失败第一时间收到通知」。

**Functional Requirements:**

#### FR-19: 多渠道通知
支持通知渠道:企业微信、钉钉、飞书、邮件,以及**自定义出站 webhook**(平台向用户配置的任意 HTTP 端点 POST 通知负载,供其它平台接入)。

**Consequences (testable):**
- 配置钉钉机器人 → 对应事件发生时该渠道收到通知。
- 配置自定义出站 webhook URL → 平台 POST 通知负载到该 URL,可被其它平台接收。

#### FR-20: 通知事件与细粒度路由
可通知事件:构建成功 / 构建失败 / 部署成功 / 部署失败 / 回滚发生 / 健康检查失败。按项目细粒度配置「哪些事件 → 发到哪些渠道」。

**Consequences (testable):**
- 配置"失败→钉钉+邮件、成功→仅钉钉" → 失败事件发钉钉与邮件,成功只发钉钉。
- 未配置路由的事件不发送通知。

#### FR-21: 通知内容自定义
通知内容支持自定义(模板 + 变量,如项目名、分支、commit、状态、耗时、错误摘要)。

**Consequences (testable):**
- 自定义模板引用变量 → 实际通知渲染为对应字段值。
- 未自定义时使用平台默认模板。

### 4.6 AI 运行时运维
**描述**:把 AI 做进失败排查与运行时观测 —— 构建/部署失败时给出根因与修复建议,运行中服务按可配置规则做异常检测并告警。这是产品最锋利的差异点(市场空地)。v1 为 **agentless**:AI 失败分析基于平台已有日志,异常检测经 SSH 轮询取数;实时流与内网穿透所需的可选 Go agent 延后(见 §安全 NFR 与 §5 Non-Goals)。AI 实现采用**现成强 LLM + 精选错误知识库(不自训模型)**;输出以**"根因假说 + 证据"**呈现而非"结论",避免虚假确定性误导。实现 JTBD「构建失败时 AI 直接指出问题」。

**Functional Requirements:**

#### FR-22: AI 失败分析
构建/部署失败时,平台调用配置的 LLM(+ 精选错误知识库),基于失败日志产出**根因假说(因果链,而非孤立的错误条目)+ 可执行修复建议 + 置信度**,展示在运行详情页,并随失败通知(§4.5)一并推送。"AI 认为"与"原始日志证据"分栏呈现,语言上为"最可能的根因是 X"而非"根因是 X"。

**Consequences (testable):**
- 构造一次依赖缺失导致的构建失败 → 详情页给出根因假说(指向该依赖)+ 修复建议 + 置信度,并附可展开的日志证据;失败通知包含该摘要。
- AI 同时显示置信度;低置信时明确提示存在其它可能根因。
- LLM 不可用 → 流水线结果照常记录,AI 分析降级为"不可用"提示,不阻断结果。

#### FR-25: 成功/失败差异对比视图
对一次失败的运行,提供其与**上一次成功运行**的差异对比视图:并排呈现变更的环境变量、配置、commit/文件,辅助快速定位"这次和上次差在哪"。

**Consequences (testable):**
- 一次失败运行存在上一次成功基线 → 差异视图并排列出变更的环境/配置/commit/文件。
- 无成功基线 → 提示"无可对比基线"。

#### FR-26: 诊断反馈闭环
用户可对 AI 根因假说标记"有用 / 无用"(可附正确根因);反馈被记录,用于改进后续诊断质量并积累平台专有的诊断数据。

**Consequences (testable):**
- 对一条根因假说点击"有用/无用" → 反馈被持久化记录,关联该运行与诊断。
- 反馈数据可供后续检索/改进引用(具体机制留架构阶段)。

#### FR-23: 可配置运行时异常检测
平台经 SSH 轮询目标服务器的服务状态/指标;**异常检测规则可按服务/框架配置**(不同服务、前后端框架的检测内容不同)。检测到异常 → 触发 §4.5 通知告警。

**Consequences (testable):**
- 为某服务配置"重启循环"规则 → 该服务反复重启时产生告警并经配置渠道通知。
- 轮询间隔可配置。
- `[ASSUMPTION]` v1 异常检测数据源为 SSH 轮询;实时流/内网穿透待 agent(延后)。

#### FR-24: 可配置 AI 提供商
LLM 提供商与密钥由用户配置,**不写死在代码**;支持多家(如 Claude / OpenAI / 本地 Ollama)。

**Consequences (testable):**
- 配置 Claude key → AI 功能走 Claude;切换本地 Ollama → 走本地模型。
- 未配置 AI → AI 功能优雅禁用,核心 CI/CD 功能不受影响。

**Notes:** 可选 Go agent(实时指标/日志流 + 内网穿透)**延后至安全评审里程碑**,届时须满足硬性安全要求(只出站拨号、mTLS 双向认证、最小权限只读、无命令执行通道、签名发布、最小依赖)—— 见 §8 安全 NFR 与 §5 Non-Goals。

## 5. Non-Goals (Explicit)

v1 明确不做(以守住"轻量、个人优先、小而锋利"):
- **多用户与 RBAC / 权限分级** —— v1 单用户登录,不做企业级权限。
- **高可用 / 集群编排** —— 不自建编排。**k8s 委托延后出 v1**(client-go 体积威胁 ≤100MB + 复杂度);v1 部署仅普通 Docker/SSH 目标。`[NON-GOAL for MVP]`
- **一键服务市场 / PaaS 化**(不长成 Coolify 那类)。
- **自带镜像仓库** —— 一律对接外部 registry(FR-7)。
- **常驻 Go agent** —— 延后至安全评审里程碑;v1 全程 agentless(SSH)。`[NON-GOAL for MVP]`
- **内嵌代码编辑/提交** —— 代码浏览只读(FR-4)。

## 6. MVP Scope

### 6.1 In Scope
- 全部 Features §4.1–§4.6 的 FR-1 ~ FR-26(**agentless**:经 SSH,无常驻 agent)。
- 平台:单管理员登录、单一 Web UI、双运行模式(Docker / 本地编译原生),常驻内存 ≤ 100MB。
- 敏感操作审计日志;凭据加密存储。
- 首发代码平台 Gitee;通知渠道企业微信/钉钉、飞书、邮件 + 自定义出站 webhook;AI 提供商可配置。

**全部 FR-1~24 均为 v1 交付目标;下述"落地排序"是实现顺序,而非范围分期。**

**落地排序(dogfooding 优先)**:先打通端到端竖切 —— Gitee push webhook → 隔离构建 → 部署到 1 台服务器 → 企业微信/钉钉通知,让作者第 1 个服务跑起来;再横向铺开其它代码平台、多机、diff、其它渠道、AI 失败分析与异常检测。借多个 AI 开发 agent 并行实现。

### 6.2 Out of Scope for MVP
- 常驻 Go agent 与内网穿透、实时指标流 —— 延后至安全评审里程碑(届时硬性安全要求见 §8)。`[NOTE FOR PM] 异常检测的实时性与内网可达性在此之前受 SSH 轮询限制。`
- 多用户 / RBAC、HA/集群、**k8s 委托部署**、服务市场、自带 registry、代码编辑 —— 见 §5。
- Discord / Slack 通知渠道 —— v1 暂不纳入。

**Primary**
- **SM-1 作者自用且稳定运营**:作者将自己约 20 个服务与网站全部迁移到 Pipewright 并稳定运营,使其成为日常部署唯一工具。验证 FR-1~FR-24 的整体可用性。
  - 里程碑:**M1** 1–2 个服务日常在用 → **M2** 20 个全部迁移、稳定不救火。

**Secondary**
- **SM-2 被看见**:发布后首年内 GitHub ≥ 100 star。
- **SM-3 被陌生人用**:发布后首年内出现非作者的真实用户 / issue。验证产品叙事与可上手性。

**Counter-metrics (do not optimize)**

- **SM-C2 安全敏感面**:不得为"求快/上线"牺牲凭据加密、登录认证、审计日志等基本防护 —— 抵消 dogfooding 速度压力。

## 8. Cross-Cutting NFRs

### 8.1 平台形态与运行(覆盖 ① Web UI、⑪ 双运行模式)
- **单一 Web UI**:所有能力经一个干净、直观的 Web 界面操作(对 Jenkins 难用 UX 的直接回应)。
- **双运行模式**:平台既可作为 Docker 容器运行,也可本地编译为原生程序直接运行(不依赖 Docker)。
- **技术栈**:Go(实现取向见 addendum)。

### 8.2 性能 / 体积预算
- 平台常驻内存 **≤ 100MB**(对标 Woodpecker,远低于 Jenkins 500MB+)。
- 构建在隔离容器内执行,不计入平台常驻内存,且不污染宿主机。

### 8.3 安全(本产品最关键的横切关注)
- **认证**:Web UI 必须登录;v1 单管理员账号(无多用户/ RBAC)。界面不得在网络上裸奔。
- **凭据**:仓库凭据、服务器 SSH 凭据、镜像仓库凭据、AI 提供商密钥等一律加密存储,**永不以明文出现在日志或界面**。
- **审计日志**:敏感操作(部署、回滚、进容器执行命令、改凭据、改服务器、改流水线配置)记录"谁、何时、做了什么"。
- **远程命令面**:FR-18 容器内执行与各类部署操作受认证约束并纳入审计。
- **未来 agent 硬性安全要求**(上线前必须满足):只出站拨号(不开入站端口)、mTLS 双向认证、最小权限只读、无命令执行通道、签名发布、最小依赖。
- **代码审核策略**:起步快速迭代,审核闸口绑定"外部用户开始接入"时点 —— 详见简报 addendum 风险章节。其中 **SSH / 密钥 / 凭据处理这一安全敏感面,建议即便在 v1 也纳入安全审查**。

**v1 安全不可妥协清单(设计期就要做对,事后补不回来;总成本约 2 人天):**
- **AC-SEC-01 凭据存储**:SSH 私钥等凭据必须存入加密保险库(Vault KV / SOPS / age),严禁明文落库、落环境变量或日志。验收:日志目录 grep 私钥为空;DB dump 无 PEM 字段。
- **AC-SEC-02 命令执行**:`docker exec` / SSH 执行必须用 exec 数组(非 shell 字符串拼接),容器/主机经白名单校验。验收:传入 `; id` 返回 4xx 且不执行。
- **AC-SEC-03 审计不可篡改**:每次 SSH/exec 操作同步写入 append-only 外部 sink(远程 syslog / 对象存储);本地 root 删本地日志后,外部记录仍完整。
- **AC-SEC-04 secret 脱敏**:运行日志落盘前对已登记的 secret 做 masking。验收:job 中 echo secret,日志输出为 [MASKED]。

### 8.4 可靠性
- 部署失败可回滚(FR-11),健康检查门控切换(FR-12);多机部署结果独立、互不连累(FR-13)。
- LLM 不可用时 AI 功能优雅降级,不阻断核心 CI/CD(FR-22/FR-24)。
