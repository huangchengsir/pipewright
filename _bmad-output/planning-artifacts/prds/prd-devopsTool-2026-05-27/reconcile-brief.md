# Brief → PRD 对账findings

对账方法:逐条比对 brief.md + addendum.md 与 prd.md,重点捕捉 FR/Feature 结构容易"静默丢失"的定性内容 —— 定位、语气、"feel"、差异化叙事、风险姿态。每条标注严重度(高/中/低)与建议动作。

---

## A. 定位 / 叙事 / "feel" 类(FR 结构最易丢失)

### A1. 「运行时 AI 运维」是"市场空地 / 最锋利差异点"的强叙事被弱化
- **Brief 说**:反复强调 AI 两条腿中"运行时智能运维"是"目前市场空地,也是最锋利的差异点"(Solution / What Makes This Different / Vision 三处),且 addendum 风险章把"差异化须真正落在运行时 AI 运维这块空地,否则泯然于众多轻量 CI"列为成败关键。
- **PRD 覆盖**:§4.6 描述句有一句"这是产品最锋利的差异点(市场空地)",但仅一处轻描;并未把它提升为产品级战略主张或贯穿性原则,FR-22/23 沦为普通功能条目。
- **严重度**:中
- **建议动作**:在 §1 Vision 或新增"差异化主张"小节中,把"运行时 AI 运维 = 唯一真正的护城河 / 不可让步的差异点"作为产品级断言写明,并与 SM-C 类反向指标挂钩(防止它被砍成"锦上添花")。

### A2. 「一个工具替掉三个」的核心差异化框架未被显式命名
- **Brief 说**:What Makes This Different 开篇定义 "核心差异化 = 一个工具替掉三个 + 轻量 + AI",并明确"没有任何现有工具同时具备 轻量+多版本隔离构建+多机部署+多机管理+AI"。这是产品的一句话立身之本。
- **PRD 覆盖**:§1 Vision 复述了"把三个工具收进一个",但未把"同时具备五要素、市场无人做到"这一**竞争性断言**保留下来。FR 列表把五要素拆散,读者看不到"五合一才是卖点"的整体性。
- **严重度**:中
- **建议动作**:在 Vision 或 MVP Scope 顶部加一句显式的差异化公式 + "五要素同时具备"的竞争断言,锚定为什么不能随意砍掉任一要素。

### A3. 竞品坐标 / 市场地形完全丢失
- **Brief 说**:给出明确竞品坐标 —— Coolify 56K★(PaaS,非同类)、Woodpecker ~6.5K★(缺多机部署 UX)、Drone ~31K★(被 Harness 收购前景不明)、Kamal/Ansible(只管部署)、ForgeAI(仅 Jenkins 运行时 AI 插件)。addendum 还补 Dokploy 34K★。
- **PRD 覆盖**:几乎为零。仅 §2.2 / §5 反复提到"不长成 Coolify 那类",其余竞品(Woodpecker/Drone/Dokploy/ForgeAI/Kamal)及其定位差距均未出现。
- **严重度**:中
- **建议动作**:在 PRD 增设极简"竞争地形"段落或附录,保留竞品坐标与"它们各缺什么",作为范围决策与差异化主张的依据(尤其 ForgeAI 证明"运行时 AI 是空地"的论据不应丢)。

### A4. 产品语气 / 情感动机("源自真实痛点、对 Jenkins 的厌烦")被抽干
- **Brief 说**:全篇有强烈的第一人称叙事与情绪 —— "苦于它的臃肿"、"繁琐的 UX"、Jenkins "太重/难用/没有 AI/能力割裂"。这是开源叙事的情感底色。
- **PRD 覆盖**:§1 Vision 保留了"嫌 Jenkins 过重、体验繁琐、毫无 AI",语气已比 brief 克制但基本在;§2.1 JTBD 也保留了用户视角。属于轻微稀释,可接受。
- **严重度**:低
- **建议动作**:可不改;若希望 PRD 也能承载开源叙事,可在 Vision 末尾保留一句"作者把 20 个生产服务托付给它"的叙事钩子(见 A5 关联)。

### A5. "连作者都把 20 个生产服务托付给它"= 最硬开源叙事 —— 叙事价值未点明
- **Brief 说**:Success Criteria 明确"『连作者都把 20 个生产服务托付给它』既是成功判定,也是最硬的开源叙事";addendum 利好牌面再次点名为"天然开源叙事"。
- **PRD 覆盖**:§1 与 SM-1 保留了"20 个服务=验收线 + 最有说服力的叙事",这一点覆盖较好。
- **严重度**:低
- **建议动作**:无需改动,已覆盖。

## B. 风险姿态 / 安全姿态类

### B1. 「逆风明显」的诚实风险姿态被 PRD 略去
- **Brief/addendum 说**:可行性判断明确二分 —— 按作者自定义标准"大概率可达",但"作为爆款开源逆风明显",并列出四大障碍:① 赛道挤获客难;② 单人维护+审核滞后的信任/安全顾虑(工具持目标机 SSH,攻击面大);③ 范围大单人易停半成品;④ 差异化须真正落在运行时 AI。
- **PRD 覆盖**:PRD 几乎不谈"风险/逆风"。§8.3 处理了安全的实现层,但"赛道挤、单人易半成品、获客难"这些战略风险姿态完全缺席。FR 结构天然不承载风险叙事。
- **严重度**:中
- **建议动作**:PRD 可不必复制全部风险,但应在文首或 Open Questions 旁加一句指针"战略风险与逆风评估见 addendum 风险章",避免下游误以为"无风险"。当前 §0 只指向 addendum 的"技术选型/架构/风险细节",可强化。

### B2. 范围风险:"单人易停在半成品" → 应反映为强落地排序纪律
- **Brief 说**:addendum 把"范围大、单人易停在半成品"列为关键障碍;brief 用"落地排序(dogfooding 优先)+ 端到端竖切"来对冲。
- **PRD 覆盖**:§6.1 保留了竖切落地排序,覆盖良好;但未把它与"防半成品"这一**风险动机**关联,读者可能当成普通排期而非风控手段。
- **严重度**:低
- **建议动作**:在 §6.1 落地排序处补一句动机说明:"竖切优先是为对冲单人项目易停在半成品的风险"。

### B3. 代码审核闸口的时点定义 —— 教练 caveat 部分丢失
- **Addendum 说**:① 审核闸口应绑定"外部用户开始拿它连自己的服务器"这一**时点**,而非"v1/v2 做完"的日历点;② SSH/密钥/凭据这一安全敏感面**即便 v1 也建议纳入审查**(事后全量补审更难);③ 作者宜保持"能读懂、能判断"的代码水平。
- **PRD 覆盖**:§8.3 保留了"审核闸口绑定『外部用户开始接入』时点"(覆盖①),并指向 addendum。但 caveat ②(SSH/凭据面 v1 即审)与 ③(作者保持判断力)**未进 PRD**。其中 ② 是可落地的安全约束,丢失风险较高。
- **严重度**:中
- **建议动作**:把 caveat ② "SSH/密钥/凭据处理面即便 v1 也纳入安全审查"提升为 §8.3 的一条明确 NFR(而非仅靠 addendum 指针)。caveat ③ 可留在 addendum。

### B4. v2 agent 的"信任面收窄"动机 vs PRD 的"硬性安全要求"——动机层丢失
- **Addendum 说**:v2 可选 agent 的买点除内网穿透/实时流外,还包括"本地自治、**缩小中控信任面**";形态明确为"独立进程/独立二进制,经 gRPC/HTTP/WebSocket 通信,参考 Drone server+agent,不是 in-process 插件"。
- **PRD 覆盖**:§4.6 Notes 与 §8.3 保留了 agent 的**硬性安全要求**(只出站、mTLS、最小权限只读、无命令通道、签名发布、最小依赖),覆盖很好;但"缩小中控信任面"这一**设计动机**与"独立进程/非 in-process、参考 Drone"这一**形态约束**未进 PRD(留在 addendum 可接受,但形态约束对架构阶段重要)。
- **严重度**:低
- **建议动作**:在 §9 Open Question 2(agent 设计)旁补一句指针,确保架构阶段不把 agent 设计成 in-process 插件;或在 PRD 明记"agent 为独立进程,经 gRPC/HTTP/WS 通信"。

## C. 范围 / 技术选型类(多为已覆盖,少量遗漏)

### C1. Go 选型的"理由逐条对齐"与"Node fallback"决策被简化为一句"Go"
- **Brief/addendum 说**:addendum 用整节论证 Go(轻量~10–50MB、单静态二进制契合双运行/零污染、Docker SDK 即 Go、同赛道正统、AI 写码使 Java 背景优势作废),并明确保留 **Node/TypeScript 作为 fallback**(非首选),否决 Java/Rust 各有理由。
- **PRD 覆盖**:§8.1 仅写"技术栈:Go(实现取向见 addendum)",指针存在。Node fallback、否决理由不在 PRD —— 这本属下游/架构内容,指针处理合理。
- **严重度**:低
- **建议动作**:无需改动;指针已足够。架构文档应继承 addendum 的选型论证。

### C2. 平台常驻内存目标:brief 文本与 PRD 不完全一致(潜在矛盾)
- **Brief 说**:What Makes This Different 写"目标常驻内存对标 Woodpecker(约 **50–100MB**)";addendum 写"Go 常驻内存 **~10–50MB**,对标 Woodpecker"。brief 内部已存在 50–100 vs 10–50 的口径差。
- **PRD 覆盖**:PRD §8.2 / SM-C1 统一定为 **≤100MB**。这是对 brief 两个口径取了上界,属合理收敛,但严格说与 addendum 的"~10–50MB"更激进的目标存在松动。
- **严重度**:低
- **建议动作**:确认 ≤100MB 为正式预算(更宽松、更安全)。若想保留 addendum 的进取目标,可注明"理想区间 10–50MB,硬上限 100MB"。无需大改。

### C3. 目标用户的"homelab / 5–50 人小团队待验证延伸"——已覆盖
- **Brief 说**:Who This Serves 把"是否延伸到 5–50 人小团队 / homelab"标为待验证。
- **PRD 覆盖**:§2.2 脚注 + §9 Open Question 1 均保留,覆盖良好。
- **严重度**:低
- **建议动作**:无需改动。

### C4. 里程碑 M1–M4 的"被看见/被陌生人用"叙事 —— 已覆盖
- **Brief 说**:M1 自用 → M2 全迁 → M3 100 star(作者认为"不错")→ M4 出现陌生人用户/issue。
- **PRD 覆盖**:SM-1(M1/M2)、SM-2(100 star)、SM-3(陌生人)均保留。M3 的"作者认为达到即不错"这一**期望管理语气**略丢失(无伤大雅)。
- **严重度**:低
- **建议动作**:无需改动。

## D. PRD 新增 / 超出 brief 的内容(对账反向检查,非 gap 但需确认)

> 以下是 PRD 引入而 brief 未明说的内容,需确认是否为作者意图(防止 PRD 单方面扩张范围)。

### D1. PRD 引入了 brief 未提的若干具体能力
- **新增**:FR-7 外部 registry 的"平台不自带 registry"显式化(brief 未提 registry 策略);FR-11 自适应部署策略(k8s 委托 vs 普通机回滚)—— brief 仅说 SSH 部署,未提 k8s 委托;FR-18 容器内交互终端(brief 未提);FR-19 自定义出站 webhook(brief 通知部分未列);FR-10 JAR/systemd、dist/nginx 的具体部署方式(brief 未细化)。
- **判断**:这些多为 PRD 阶段合理的具体化,方向与 brief 一致。但 **k8s 委托(FR-11)** 与 **容器内交互终端(FR-18)** 是范围扩张,且后者是高安全敏感面 —— brief"明确不做"里强调"小而锋利",需确认这两项是否在作者意图内。
- **严重度**:中(范围蔓延风险,与 brief"小而锋利"姿态可能张力)
- **建议动作**:向作者确认 FR-11 k8s 委托与 FR-18 容器内终端是否纳入 v1;若纳入,FR-18 需明确受 §8.3 审计/认证约束(PRD 已部分处理)。

## 汇总(按严重度)

**中(应处理)**
- A1 运行时 AI 运维的"最锋利差异点/市场空地"叙事被弱化
- A2 "一个工具替掉三个 + 五要素同时具备"的差异化框架未显式命名
- A3 竞品坐标 / 市场地形完全丢失
- B1 "逆风明显"的战略风险姿态在 PRD 缺席
- B3 审核 caveat②(SSH/凭据面 v1 即审)未进 PRD
- D1 FR-11 k8s 委托 / FR-18 容器终端 的范围扩张需确认

**低(可选 / 已覆盖)**
- A4 语气稀释(可接受)、A5 已覆盖
- B2 竖切排序的风控动机可补一句、B4 agent 形态约束建议加指针
- C1/C2/C3/C4 多为已覆盖或合理收敛

**总体**:PRD 对 brief 的**功能性内容**继承良好,FR 完整且更具体。主要损失集中在**定性叙事层**(差异化主张、竞品地形、运行时 AI 的战略地位)与**风险姿态层**(逆风评估、审核 caveat),以及少量**范围扩张需回确认**(k8s/容器终端)。建议优先补 A1/A2/A3(差异化叙事)与 B3(可落地安全约束),其余以指针指向 addendum 即可。
