# PRD 质量评审 — Pipewright（2026-05-27）

## 总体结论（Gate verdict）

**PASS-WITH-FIXES。**

这是一份高于平均水准的单人/开源项目 PRD:有真实论点(把 CI 工具 + Ansible/Kamal + Portainer 三件套收进一个轻量 Go 工具,并把 AI 做进运维),有量化的成败标准(承载作者全部约 20 个生产服务),术语表锚定且大体被遵守,FR 普遍带可测后果(Consequences),Non-Goals / 反向指标 / 开放问题都不是摆设。论点—特性—指标的链条基本闭合,可以据此进入架构与故事拆分。

需要修复后才算干净的点集中在三处:**(1)** 术语表把"容器"声明为"服务"的子类型,但正文 §4.4 大量以"服务/容器"并列使用,下游抽词会产生歧义;**(2)** §6.1 In Scope 把 FR-1~FR-24 全部纳入 MVP,却又在落地排序里描述了竖切优先 / 横向铺开的分期,二者对"MVP 边界"的口径不一致;**(3)** 个别 FR 后果用了"在约定时延内""尽力实现零停机"这类无界形容词,工程上无法判定 done。这些都是局部、可定点修的缺陷,不动摇整体可建性。

按本项目实际定位(单人 dogfooding + 开源,非企业流程)校准:没有把缺 RBAC、缺多租户、缺 SLA 当问题——这些已被显式列入 Non-Goals,处理得当。

## 高(High)

- **[high] MVP 边界与落地排序自相矛盾(§6.1)** — In Scope 写"全部 Features §4.1–§4.6 的 FR-1 ~ FR-24",把 24 条 FR 全列为 MVP;但紧接着的"落地排序(dogfooding 优先)"又明确分为「先端到端竖切(单服务器 / 企业微信钉钉通知)」与「再横向铺开 其它代码平台、多机、diff、其它渠道、AI 失败分析与异常检测」。读者无法判断:多机(FR-13)、diff(FR-4)、AI 失败分析(FR-22)、异常检测(FR-23)究竟是 MVP 必交付项,还是竖切之后的延伸项。SM-1 的 M1(1–2 服务在用)与 M2(20 服务全迁)其实暗示了分期,但 In Scope 的"全部 FR-1~FR-24"口径与之冲突。*Fix:* 明确二选一——要么把 In Scope 拆成「竖切必交付 FR 集」与「横向铺开 FR 集」两层并与 M1/M2 对齐;要么保留"全部 FR 为 MVP"但把"落地排序"改写为纯实现顺序声明(不暗示范围分期),并显式说明 M1 只验证竖切子集、M2 才覆盖全部 FR。

- **[high] 术语表"服务/容器"关系在正文被破坏,下游抽词歧义(§3 + §4.4 FR-15/FR-18)** — 术语表定义 **服务(Service)** = "运行实例(容器或进程)",即容器是服务的一种实现。但 §4.4 描述与 FR-15 反复用「**服务/容器层**」「运行的**服务/容器**」把"容器"与"服务"并列呈现,FR-18 标题又是「**容器**内命令执行」、正文「进入目标机**容器**」。这制造了两个问题:(a)"服务/容器"并列暗示二者是不同概念,与术语表的包含关系矛盾;(b)FR-18 直接以"容器"为操作对象,但术语表里没有独立的"容器"词条,下游(UX/架构)无法确定 FR-18 是只对容器型服务生效、还是对进程型服务也有等价终端。*Fix:* 统一口径——若"容器"是有意区分的独立概念,在 §3 增加 **容器(Container)** 词条并说明它与 **服务** 的关系(服务的一种承载形态);并把 FR-15 的"服务/容器层"改为一致表述(如"服务层(含容器型与进程型)")。FR-18 应说明对进程型服务的等价/不适用情况。

## 中(Medium)

- **[medium] 无界形容词使 done 不可判定(FR-1, FR-11)** — FR-1 后果"在**约定时延内**创建一次流水线运行"——"约定时延"未给数值或上界,工程无法测;FR-11「**尽力实现**零停机切换」——"尽力"非可验收语,且与同 FR 后果"新版本通过健康检查后切换"存在松紧不一。*Fix:* FR-1 给一个可测上界(如"webhook 收到后 ≤ Ns 内创建运行",或至少标注"目标时延待定 `[NOTE FOR PM]`")。FR-11 把"尽力零停机"改为可判定条件——例如"普通目标采用 新起→健康检查通过→切流→旧版本保留 的顺序;切换期间旧版本保持可用直至新版本就绪",删掉"尽力"。

- **[medium] FR-13 与 FR-11 对"回滚单位"的隐含口径需澄清(FR-13)** — FR-13 说多机部署"失败机按 FR-11 自行回滚,不连累成功机",但 FR-11 描述的是单目标的版本保留/回滚。当一台失败、两台成功后,整个流水线运行的"成功/失败"判定标准没有给出(部分成功算成功还是失败?是否触发 §4.5 的"部署失败/回滚"通知?发哪台的?)。*Fix:* 在 FR-13 后果中补一条:定义流水线运行级别的聚合状态(如"任一目标失败 → 运行标记为 部分失败,逐机状态独立上报,失败机触发回滚通知、成功机触发成功通知"),消除与 §4.5 通知事件的对接空白。

- **[medium] 部署成功通知缺失于 FR-20 事件集,与 JTBD/SM 主张不完全对齐(FR-20)** — JTBD 与 §4.5 强调"成功/失败第一时间收到通知",FR-20 事件集为「构建成功/构建失败/部署成功/部署失败/回滚发生/健康检查失败」——这里其实是完整的,问题在于 FR-13 引入的"部分失败/逐机"状态没有对应可通知事件。配合上一条,确认是否需要"部署部分失败"作为独立可路由事件。*Fix:* 若采纳上一条的"部分失败"聚合状态,则在 FR-20 事件枚举中显式加入,避免下游通知路由出现无法表达的状态。

- **[medium] SM-2/SM-3 缺少时间窗,弱化可验收性(§7)** — SM-1 有 M1/M2 里程碑,但 SM-2(≥100 star)、SM-3(出现非作者真实用户/issue)没有时间窗或参照点,作为开源项目"被看见"的次级指标可以接受,但严格说不可判定 done。*Fix:* 给一个宽松时间窗(如"开源后 6 个月内")或明确标为"无期限的方向性指标,不作为 gate"。

## 低(Low)

- **[low] "流水线运行"在正文常被简写为"运行/一次运行"(§3 + 全文)** — 术语表词条为 **流水线运行(Pipeline Run)**,FR-1 等处简写为"一次运行""创建运行"。语义可解析,但严格违反 §3"不得引入同义词"的自我约束。*Fix:* 首次出现用全称,后续简写可接受;或在术语表注明"运行"为 流水线运行 的允许简写。

- **[low] §8.1 标题里的"覆盖 ① Web UI、⑪ 双运行模式"圈号编号无来源锚点(§8.1)** — ①/⑪ 像是引用某份编号清单(可能是 brief),但 PRD 内无对应编号表,读者无法解析这两个圈号指向。*Fix:* 删除圈号编号,或注明其来自 brief 的能力清单编号并给出引用。

- **[low] Assumptions Index 与内联 `[ASSUMPTION]` 往返一致(校验通过,记录备查)** — 内联两处 `[ASSUMPTION]`(FR-11 保留 1 份、FR-23 SSH 轮询数据源)均在 §10 索引中有对应条目,反之亦然。无遗漏。`[NON-GOAL for MVP]`(§5 agent)与 `[NOTE FOR PM]`(§6.2)也都落在真实张力点上,非安全检查点。此项无需修复。

- **[low] FR-9 校验"目标服务器存在"与 FR-14 登记时点的次序未明(FR-9)** — FR-9 要求保存流水线配置时校验"引用的目标服务器存在",隐含要求服务器先于流水线配置登记。属合理约束但未明说依赖次序。*Fix:* 可在 FR-9 或 §4.1 描述中一句话点明"流水线配置引用的服务器须已在 §4.4 登记",便于下游建模依赖。

## 机械性核对(Mechanical notes)

- **FR ID 连续性**:FR-1 ~ FR-24 连续、唯一、无缺号无重复。✅
- **特性—JTBD 追溯**:§2.1 的 5 条 JTBD 均能映射到特性——push 自动更新→§4.1/§4.3;多版本隔离→§4.2;一处看多机→§4.4;成功失败通知→§4.5;失败 AI 指出→§4.6。每个 §4.x 描述末尾都显式回指对应 JTBD,追溯链清晰。✅
- **SM—FR 追溯**:SM-1 显式"验证 FR-1~FR-24 整体可用性";反向指标 SM-C1/SM-C2 分别锚定 §8.2 体积预算与 §8.3 安全 NFR。✅ 闭合良好。
- **Glossary 漂移**:主要问题为"服务/容器"并列(见 High);"流水线运行"简写(见 Low)。**构建产物**、**流水线配置**、**凭据**、**镜像仓库**、**目标服务器**等术语全文一致,无同义词漂移。
- **交叉引用解析**:§4.x 之间的 "见 §4.3 / 见 §8 安全 NFR / 见 §5 Non-Goals / FR-3 / FR-11" 等引用均能定位到真实小节/FR。§8.1 的圈号 ①⑪ 是唯一无法解析的引用(见 Low)。
- **NFR 完整性**:§8 覆盖平台形态、性能/体积(≤100MB 有具体阈值,非套话)、安全(认证/凭据/审计/远程命令面/未来 agent 硬性要求,具体且产品特定)、可靠性(回滚/健康检查/降级)。**缺一项常见维度:无明确的数据持久化/备份与平台自身升级/迁移 NFR**——单人工具可暂缓,但建议作为 Open Question 记一笔(平台自身配置/凭据库如何备份与迁移)。
- **开放问题**:§9 五条均为真开放问题(目标用户边界、agent 上线时点、k8s 委托边界、回滚保留份数、异常检测模板覆盖),与内联假设/Non-Goals 呼应,无"自问自答"式伪开放。✅
- **Shape fit**:产品为单操作者内部工具 + 开源叙事。PRD 采用 capability-spec 形态(FR 为主、JTBD 替代重型 UJ、SM 偏运营),与产品形态匹配,未被过度形式化(没有给单操作者强加多 persona/UJ),也未欠形式化。✅
