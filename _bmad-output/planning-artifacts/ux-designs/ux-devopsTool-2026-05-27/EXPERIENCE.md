# devopsTool — Experience (EXPERIENCE.md)

> 本文件是 devopsTool 的**体验契约**:信息架构、行为、状态、交互、可访问性、关键流程。视觉身份(颜色/字体/间距/组件外观)在 `DESIGN.md`,本文以 `{path.to.token}` 语法按名引用其令牌。**两份脊柱(本文 + DESIGN.md)在与任何 mock / 线框 / 导入冲突时为准。** 术语严格沿用 PRD §3 术语表(项目 / 流水线配置 / 流水线运行 / 环境 / 目标服务器 / 凭据 / 构建容器 / 构建产物 / 镜像仓库 / 服务 / 容器 / 服务器分组 / 通知渠道),不引入同义词。

## Foundation

单一 Web 界面,**Web 桌面优先**(运维控制台,主要在笔记本/台式机上使用;窄屏可读不可主用)。**单管理员**,无多用户、无 RBAC、无团队切换 —— 界面里永远只有"你",没有归属/权限的二义。

技术基座(承自架构):**Vue 3 + Vite + Pinia + Vue Router**,组件库 **Naive UI**(轻、现代,契合"干净直观、反击 Jenkins 的繁琐 UX")。本文只规定 Naive UI 默认之外的**行为增量**;凡 Naive UI 已有合理默认(焦点环、键盘交互、弹层行为)即继承。实时数据经 **SSE** 单向流(日志 tail、运行状态、服务器指标);**WebSocket 仅用于容器交互终端**(FR-18)。代码浏览/diff 用 **Monaco 只读、懒加载**(FR-4/FR-25)。整个前端 `go:embed` 进单二进制,双运行模式(原生 / Docker)对体验无差异。

`DESIGN.md` 是视觉身份参考:深色为默认主题、浅色为对等主题(双令牌切换);单一品牌色**电青**({colors.primary} / {colors.cyan})+ 语义状态色(绿 {colors.green} = 成功、红 {colors.red} = 失败、琥珀 {colors.amber} = 进行中/警告);Inter(无衬线)+ JetBrains Mono(等宽,用于 ID/分支/commit/日志/指标);动效贯穿但只用合成器友好属性,并遵守 `prefers-reduced-motion`。**禁用"AI 套路"图标**(✦ 星芒等):AI 能力一律用克制的工程/诊断标记(活动脉冲折线)+ 青色语义表达,不靠装饰卖弄。

## Voice and Tone

微文案。品牌语气与美学姿态在 `DESIGN.md.Brand & Style`。devopsTool 对工程师说话:**直、准、用动词与数字,不卖萌、不报喜、不堆惊叹号**。AI 永远以"假说"语气说话,绝不冒充结论。

| Do | Don't |
|---|---|
| 「部署到 生产-1 失败」 | 「哎呀,出错了 😢」 |
| 「最可能的根因是 `DB_PASSWORD` 未注入(假说,非结论)」 | 「根因:DB_PASSWORD 未注入」(伪确定) |
| 「2 台成功 · 1 台已回滚」 | 「部署完成!」(掩盖部分失败) |
| 「LLM 提供商无响应,本次未生成诊断。运行结果与日志照常记录,核心 CI/CD 不受影响。」 | 「AI 服务异常」(不说后果,徒增焦虑) |
| 「首次部署无可对比基线,差异视图暂不可用。」 | 「No baseline」 |
| 「经 SSH → docker exec,操作纳入审计」(进终端前的 tooltip) | 静默执行高危操作 |
| 「输入 `acme` 确认永久重置」 | 「确定吗?」 + 一个红按钮 |
| 凭据一律以掩码/handle 呈现:`sk-••••a91f` | 任何位置出现明文密钥 |

语义词汇**固定不可换词**(运行/服务/项目通用):成功 · 失败 · 进行中 · 部分失败 · 已回滚 · 排队中。

| 流水线画布 + drawer | 流水线编排 | 任务卡点选 → 右侧 drawer 配置(就地,不弹模态、不跳页)。drawer 随选中任务换内容;未选时收起。 |
| 结果通知 drawer | 流水线编排(选中「结果通知」任务) | 顶部范围分段「继承全局默认 ↔ 为本流水线自定义」;切到自定义才展开本流水线事件路由 chip 矩阵 + 模板来源 + 额外接收人。见 §产品模型「通知」。 |

| 服务行 | 服务器详情 | 名称/镜像/端口/状态 + 操作组:重启 ⟳ / 日志 ▤ / **进入终端 ›_**。重启等破坏性操作走二次确认。 |
| 容器终端 | 服务行「›_」(FR-18) | WebSocket 交互终端;**进入前 tooltip 明示「经 SSH → docker exec,操作纳入审计」**;命令以 exec 数组执行(AC-SEC-02),目标白名单。 |
| 日志终端 | 运行详情 / 服务日志 | 等宽、级别着色(INFO/WARN/ERROR)、关键词高亮、SSE 标识、跟随尾部开关、自动滚动。历史回看走时间范围。 |
| 置信度环/标记 | 诊断卡内 | 圆环/条按百分比填充(动画);高(≥75%)/中/低三档;低置信时明确提示存在其它可能根因。 |
| Toast | 全局结果反馈 | 右下角堆叠;成功/信息 4s 自动消失(底部进度条);**错误手动关闭**;可带 1 个行动(如失败 toast 带「查看 AI 诊断 →」)。四型:成功/错误/警告/AI 信息(青)。 |
| 内联 Banner | 页面/卡片内常驻提示 | 不自动消失;语义底+边(`*-soft`/`*-line`);用于"无基线""未配健康检查""凭据错误""校验通过"等情境提示。 |
| 二次确认对话框 | 破坏性操作 | 普通(回滚/重启/删凭据):标题+后果说明+取消/确认。**高危(重置实例/删服务器):type-to-confirm** —— 须输入实例名/服务器名,确认按钮在匹配前禁用。 |
| 高频操作模态 | 添加服务器/手动触发/添加凭据/添加通知渠道 | 统一语言:图标+标题/副标题、字段网格、底部「次按钮 + 主按钮」、测试类操作(测试连接/发送测试)置左下、危险操作色异化。模态栈不超过 1 层。 |

## State Patterns

| 状态 | 出现处 | 处理 |
|---|---|---|
| 冷加载 | 列表/详情数据加载 | **骨架屏**(shimmer 1.4s)匹配预期布局,**不用转圈遮罩**。 |
| 空 · 无项目 | 项目列表 / 概览 | 图标 + 「还没有项目」+ 「接入第一个 Gitee 仓库,AI 会分析代码帮你生成流水线配置。」+ 主 CTA「+ 新建项目」。 |
| 空 · 首次使用 | 登录后无任何配置 | 进入引导(`mock-onboarding.html`):价值三卡 + 3 步清单(连 AI ✓ / 加服务器 / 建项目),步骤间锁依赖;可「跳过引导」,引导项随时可在设置重开。 |
| 局部加载 | 按钮提交中 | 主按钮内联 spinner + 禁点;不阻塞全屏。 |
| 进行中(长任务) | 运行详情 | SSE 推送状态 + 实时日志;确定进度用进度条,不确定用流光/spinner。 |
| 字段级错误 | 表单 | 字段内联红边 + 3px 红光环 + 字段下红字说明(如「Webhook 地址格式不正确」)。服务端权威校验(FR-9);前端镜像校验仅为体验。 |
| 整页加载失败 | 任意数据屏 | 居中错误块 + 原因(如「无法连接到 生产-1(SSH 超时)」)+ 「↻ 重试」。 |
| **AI 诊断不可用(降级,FR-22)** | 失败运行诊断区 | 青色虚边降级卡 + 「暂不可用」标:「LLM 提供商无响应,本次未生成诊断。**运行结果与日志照常记录,核心 CI/CD 不受影响。**可前往设置检查 AI 提供商,或稍后重试诊断。」**绝不阻断运行结果落库或通知。** |
| AI 诊断生成中 | 失败发生后短暂 | 诊断卡占位骨架 + 「诊断生成中…」;失败结果与日志已先行可见。 |
| 凭据错误 | 拉仓库/连服务器/推镜像 | 明确「凭据错误」失败态;banner「请检查 Gitee 访问令牌 · 前往凭据 →」;**日志与界面均无凭据明文**(AC-SEC-01);已登记 secret 在日志落盘前及发往 LLM 前均脱敏为 [MASKED](AC-SEC-04)。 |
| 无基线 | diff / 成功失败对比 | banner「首次部署无可对比基线,差异视图暂不可用。」不渲染空 diff(FR-4/25)。 |
| 未配健康检查 | 部署配置/运行 | 警告 banner:「未配置健康检查 —— 容器启动成功即视为部署成功。」(FR-12) |
| 部分失败 | 多机运行 | 「部分失败」琥珀徽标 + 各台独立卡;文案强调**结果独立、互不连累**;失败台已自动回滚、线上可用(FR-13)。 |
| 已回滚 | 运行/服务 | 「已回滚」徽标 + 回滚目标版本号;说明当前线上仍可用。 |
| 凭据闲置告警 | 凭据保险库 | 闲置超期(如 60 天)的凭据标告警,提示复核(安全卫生)。 |
| 资源接近预算 | 系统设置 | ≤100MB 预算条;接近上限时变色提示(轻量是头号卖点,SM-C1)。 |

## Interaction Primitives

**鼠标优先的运维控制台**(对照 Linear 的键盘优先:devopsTool 多为低频高后果操作,可点、可读、可审计胜过手速)。

- **点击即达**:运行行/项目卡/服务行整体可点,失败行直达诊断(少一跳)。
- **就地编辑**:流水线任务配置走右侧 drawer,不弹模态、不跳页;模态仅用于"新增一个东西"(加服务器/触发/凭据/渠道)与确认。
- **二次确认分级**:可逆操作直接执行(信任用户);**破坏性**(回滚/重启/删凭据)走确认对话框;**高危**(重置实例/删服务器)走 type-to-confirm。
- **实时即推**:运行中日志/状态、服务器指标经 SSE 自动推,无需手动刷新;终端经 WS 双向。
- **跟随尾部 / 自动滚动**:日志默认跟随最新,用户上滚即暂停跟随,提供"回到尾部"。
- **主题切换**:右下角「明/暗」常驻,即时切换,记忆选择。
- **复制**:ID / commit / webhook URL / 签名密钥可一键复制;凭据复制的是引用 handle,**绝不复制明文**。
- **键盘 & 焦点**:继承 Naive UI 焦点/键盘默认;`Esc` 关闭最顶层 drawer/模态/弹层;`Tab` 序匹配阅读序。

**全局禁止**:任何位置出现凭据明文 · 用字符串拼接执行 SSH/exec(AC-SEC-02) · 模态栈 > 1 层 · 用转圈遮罩替代骨架屏 · AI 套路装饰图标(✦ 等) · 把失败结果阻塞在 AI 诊断之后(AI 永远是旁路) · 自动播放分散注意力的动效(动效须服务于流程理解)。

## Accessibility Floor

行为层。视觉对比度在 `DESIGN.md`(双主题均须达标,语义色不仅靠色相区分,辅以图标/文字)。

- 目标 **WCAG 2.2 AA**,桌面 Web 全程键盘可达。
- **状态不单靠颜色**:每个状态徽标 = 语义色 + 圆点 + 文字(成功/失败/进行中…),色盲用户凭文字与形状即可辨。
- 屏幕阅读器在导航切换时播报当前一级目的地与上下文(如「运行 #128,失败,AI 诊断」)。
- **实时区用 `aria-live`**:运行状态变化、新到日志行、诊断生成完成以 polite 区播报;高频日志节流播报,避免淹没。
- **诊断的"假说"语气对辅助技术同样成立**:置信度以文字(「置信度 88% · 高」)而非仅环形传达。
- `prefers-reduced-motion` 下:跳过入场/脉冲/环填充/流光/扫过等动效,直接呈现终态;状态切换瞬时生效,语义不丢。
- 焦点环可见(继承 Naive UI `{colors.primary}` 环,双主题均达 AA);`Esc` 永远关最顶层浮层;`Tab` 序 = 阅读序。
- 终端/日志区可键盘聚焦、可滚动;终端进入前的审计提示同时是可访问的说明文本。
- 表单错误同时以视觉(红边)+ 文字 + 可被辅助技术读取的关联(`aria-describedby`)呈现。
- 等宽字体仅用于机器文本(ID/日志/指标),不用于正文,保证可读性。

## 产品特定模型(Product-Specific Patterns)

devopsTool 有两个跨多屏、决定体验对错的核心模型,单列于此。

### 通知:全局默认 + 逐流水线覆盖(继承/覆盖为整体切换)

承自决策日志。三层分明,不可混淆:

1. **渠道连接 = 全局共享**(只配一次)。通知页的「通知渠道」卡承载企业微信/钉钉、飞书、邮件、自定义出站 webhook 的连接;所有流水线共用同一套连接。流水线**不**单独连渠道。
2. **事件→渠道路由 + 内容模板 = 全局默认**。可路由事件:构建成功 / 构建失败 / 部署成功 / 部署失败 / 回滚发生 / 健康检查失败(FR-20)。内容模板支持变量({{project}} {{branch}} {{commit}} {{status}} {{env}} {{duration}} {{ai_root_cause}} {{run_url}}),未自定义用平台默认模板(FR-21)。通知页的路由矩阵与模板标「全局默认」;**新流水线默认继承**这套规则。
3. **每个流水线可在自己的「结果通知」任务里切到"为本流水线自定义"**。范围分段是**整体切换**:选"继承全局默认"则完全跟随全局;选"为本流水线自定义"则**整套替换**本流水线的事件路由(渠道仍取自全局已连接的)+ 可选覆盖模板 + 可选额外接收人。**不是逐事件 merge** —— 不存在"某事件继承、某事件覆盖"的混合态。

通知页的「流水线覆盖」概览如实标注每个流水线是「沿用全局默认」还是「自定义路由」,并给出"在流水线中配置 →"的跳转。(`mock-notifications.html` + `mock-project-pipeline.html` 结果通知 drawer)

### AI 失败诊断:假说 + 证据 + 置信度 + 反馈闭环(旁路,绝不阻断)

承自 FR-22/25/26 与决策日志。诊断永远是**对失败的事后旁路解读**,绝不挡在运行结果与通知之前。

- **语气**:「**最可能的根因是 X**」(假说),卡上明标"假说,非结论";绝不写"根因:X"这种伪确定句。
- **三分结构**:根因假说(因果链,非孤立错误条目)· 可执行修复建议(带行动按钮,如「恢复 #126 环境」)· **置信度**(高/中/低 + 数值);低置信时明示存在其它可能根因。
- **证据并栏**:「AI 认为」与「原始日志证据」分栏;证据可展开,命中行红底 —— 让用户能自行复核假说。
- **成功/失败 diff 佐证**(FR-25):并排上一次成功 vs 本次失败的环境变量/配置/commit/文件,疑似根因项打红标。
- **反馈闭环**(FR-26):每条假说带 👍/👎(可附正确根因);反馈持久化、关联该运行与诊断,汇入设置页「诊断反馈闭环统计」(准确率/反馈数/环比),并入平台专有诊断知识库。
- **降级**(见 §State Patterns):LLM 不可用时诊断区降级为"暂不可用",运行结果/日志/通知照常。
- **随通知推送**:失败通知携带诊断摘要({{ai_root_cause}} 变量),让用户在 IM 里先看到根因再决定是否进控制台。

## Key Flows

具名主角的真实会话,各含一个高潮节拍。

### 流程 1 — 凌晨被叫醒的生产失败 → AI 诊断 + 差异对比定位 → 修复(黄佳伟,03:17,作者本人,手机收到企业微信告警)

1. 黄佳伟手机弹出企业微信:「[失败] acme-web · main · 部署 生产-1 失败 · AI 根因:`DB_PASSWORD` 未注入」。摘要已让他心里有底。
2. 他打开笔记本进 devopsTool(深色主题正合凌晨)。概览首屏「待处理失败 2 · AI 已诊断,待你处理」+ AI 洞察:「本周 3 次失败,其中 2 次同因(环境变量缺失)」。
3. 点最近运行里红色渐变的 `#128` 行(带「查看诊断」)→ **直达失败 + 诊断态**(`mock-run-diagnosis.html`),省去先进列表再翻。
4. 头部已标「已自动回滚到 #126」—— 线上没断,他不必救火,可以从容排查。
5. AI 诊断卡:「最可能的根因是环境变量 `DB_PASSWORD` 未注入 · 置信度 88% · 高(假说,非结论)」;证据栏 `app.log` L251 红底 `FATAL DB_PASSWORD is empty`;残余 12% 指向连接池。
6. 右侧成功/失败差异对比:`#126 ✓ ↔ #128 ✗`,`DB_PASSWORD` 行打「疑似根因」红标 —— 上次有、这次没了,一眼坐实。
7. **高潮**:他不需要翻一堆日志、不需要把 log 粘到 ChatGPT。AI 给的假说 + 那行红底证据 + 那一行差异,三者在同一屏对齐成一条因果链。他点诊断卡上的「恢复 #126 环境 →」,凭据补回 `DB_PASSWORD`,「重新部署」。
8. 他给这条假说点 👍(它确实对),反馈汇入诊断统计,让下次更准。

失败兜底:若此刻 LLM 恰好不可用,诊断区降级为"暂不可用",但失败结果、回滚记录、证据日志、成功/失败 diff **照常呈现** —— 他仍能靠 diff 自己定位,CI/CD 不受影响。

### 流程 2 — 首次开箱到第一个绿色部署(黄佳伟,周六上午,刚把二进制跑起来)

1. 浏览器打开实例 → 登录页(`mock-login.html`,「自托管 · 单管理员」),admin 登录。
2. 落到引导(`mock-onboarding.html`):价值三卡(≤100MB / 一个工具替三件套 / AI 失败诊断)+ 3 步清单。
3. 第 1 步「连接 AI 提供商」:他填 Claude key(存入加密保险库),测试连接通过 → ✓「已连接 Claude」;第 2 步解锁。
4. 第 2 步「添加第一台服务器」:模态里填 host + 从保险库选 SSH 私钥,平台校验 SSH + Docker 连通性(agentless,目标机零安装)→ 通过;第 3 步解锁。
5. 第 3 步「创建第一个项目」:接入 Gitee 仓库 → 进新建向导(`mock-new-project.html`)。AI 拉代码分析:「检测到 Node 22 前端项目(Vite + Vue)· 含 Dockerfile · 建议产物 Docker 镜像」,生成 4 项配置预览。
6. 他用一句话补充「push main 部署到生产」,预览实时更新映射;点「接受并创建项目」(也可逐项部分接受)。
7. **高潮**:他在新建好的流水线点「▶ 运行」。运行详情进入进行中态(`mock-run-live.html`):流水线节点逐个变绿、SSE 日志实时滚动、当前节点脉冲。几十秒后切到成功态(`mock-run-success.html`)—— 绿徽标、零停机切换 4 步流程图走完。第一个服务上线了。右下角 toast「部署成功 · acme-web #1 已上线 生产-1」。dogfooding 的第一格 M1 达成。

空态兜底:他若中途「跳过引导」,概览即是空态(「还没有项目」+ CTA),引导项随时可在设置重开。

### 流程 3 — 一次推三台,一台部分失败并自动回滚(黄佳伟,工作日 09:42,升级 Node 运行时)

1. 他 push `main`(升级 Node 运行时 `d52e8a0`)。流水线配置把 `main` 映射到 prod 的三台目标服务器,一次构建并行下发(FR-13)。
2. 运行详情进入进行中,扇出三张目标卡同时跑。生产-1、生产-2 拉镜像→起新实例→健康检查→零停机切换,各自变绿。
3. 生产-3 起了新实例,但**健康检查 30s 超时**(该机内存配额仅 256MB,另两台 512MB),切换被跳过。
4. **高潮**:整体落定为「部分失败」(`mock-run-multi.html`)。聚合条「2 成功 · 1 失败(已回滚)」。生产-1/2 已切到 `#130` 正常服务,**完全不受生产-3 牵连**;生产-3 自动回滚到 `#128`,线上仍可用。失败台卡上「健康检查超时 → 切换(已跳过)→ 已回滚 #128」,带「查看 AI 诊断」入口(AI 初判内存配额偏低)。这正是产品要证明的:多机结果独立、坏一台不连累、坏的那台自己退回安全态。
5. 部署部分失败事件已按路由发出(企业微信 + 邮件),**携带失败目标清单**。他不慌不忙点「仅重试 生产-3」,先去给那台扩内存。

失败兜底:若他想整体回退,可选「回滚全部到 #128」(走二次确认,说明将停止 #130 实例)。

### 流程 4 — 给关键流水线配自定义通知,把成功降噪(黄佳伟,午后整理告警噪音)

1. 他嫌 acme-web 每次成功都刷企业微信,想只在失败/回滚时被打扰。
2. 进通知页(`mock-notifications.html`):渠道连接是全局共享的(企业微信、邮件已连接),路由矩阵是「全局默认」。他**不**在这里改全局 —— 那会动所有流水线。
3. 「流水线覆盖」概览显示 acme-web 当前「沿用全局默认」。他点进 acme-web 流水线编排,选「结果通知」任务,右侧 drawer 出现。
4. 范围分段从「继承全局默认」切到「**为本流水线自定义**」—— 整体切换,展开本流水线事件路由 chip 矩阵。
5. **高潮**:他把"部署成功""构建成功"设为「静默」,把"部署失败""回滚发生""健康检查失败"留在企微+邮件。drawer 顶部明示「渠道连接来自全局通知设置;这里只决定本流水线哪些事件→哪些渠道」。保存。回到通知页,acme-web 的覆盖概览变成「自定义路由 · 失败/回滚 → 企业微信 + 邮件 · 成功静默」。其它流水线纹丝不动 —— 覆盖是逐流水线的整体替换,边界清清楚楚。

失败兜底:若他后悔,把范围分段切回「继承全局默认」即整体恢复全局规则,不残留半套自定义。
